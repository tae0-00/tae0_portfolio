문제 해결 기록

기능을 동작시키는 데 그치지 않고, 문제의 원인을 구조에서 찾아 해결한 작업들을 기록합니다. 문제 → 원인 → 해결

CASE 01주문 부분취소 — DB 구조 재설계집사상점 · 관리자
● 문제

주문 취소가 주문 전체 단위로만 가능해, 한 주문에 담긴 여러 상품 중 일부만 취소할 방법이 없었습니다.

▲ 원인

취소 상태를 주문 단위로만 관리해, 상품별 상태를 담을 구조 자체가 없었습니다.

✓ 해결

ORDER_DETAIL에 상품별 상태 컬럼을 추가하고, 마이페이지·관리자·배송비 귀책 로직을 전면 재구현해 데이터 정합성을 확보했습니다.

스키마 재설계상태 동적 계산(CASE)배송비 귀책 분기데이터 정합성
CASE 02외부 API 장애 대비 — Haversine 폴백전북로 · 역사 코스
● 문제

코스 동선 계산을 외부 지도 API에 의존하다 보니, API 장애 시 동선 정보가 통째로 끊길 위험이 있었습니다.

▲ 원인

정상 응답만 가정한 설계 — 외부 의존성의 실패 경로가 비어 있었습니다.

✓ 해결

호출 실패 시 두 좌표의 직선거리를 직접 계산하는 Haversine 폴백을 넣어, API가 불안정해도 동선이 끊기지 않게 했습니다.

Kakao Directions APIHaversineFallback 설계장애 대비
CASE 03라이브러리 없이 직접 구현 — GeoJSON → SVG전북로 · 메인
● 문제

행정구역 데이터를 클릭 가능한 지도로 그려야 했지만, 무거운 지도 라이브러리를 쓰기엔 과했습니다.

▲ 원인

위경도 좌표계와 화면 좌표계가 달랐습니다. 화면 y축은 위도와 반대 방향이라 그대로 그리면 지도가 뒤집혔습니다.

✓ 해결

좌표를 화면에 직접 투영하고 y축을 반전 처리해, 라이브러리 없이 클릭 가능한 인터랙티브 SVG 지도를 완성했습니다.

GeoJSON좌표 투영SVG pathNo-library
CASE 04방문자 중복 집계 방지 — 메모리 관리전북로 · 관리자 대시보드
● 문제

같은 사용자가 여러 번 접속하면 방문자 수가 부풀려져, 지표를 신뢰할 수 없었습니다.

▲ 원인

방문 이벤트를 단순 누적만 하고, 같은 사용자를 식별·중복 제거하는 구조가 없었습니다.

✓ 해결

날짜별 Set에 회원 ID·IP 식별자를 저장해 중복을 제거하고, 지난 날짜 데이터는 자동 제거해 메모리가 쌓이지 않게 했습니다.

ConcurrentHashMapSet 중복 제거메모리 관리
